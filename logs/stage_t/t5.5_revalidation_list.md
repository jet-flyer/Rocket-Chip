could not be fully live-validated during T5.5 development because the
underlying ~6% station→vehicle first-try ACK rate (the exact problem Stage
T is built to fix) prevented SET commands from landing reliably. Each item
here is "dry code validated, live round-trip verification required before
Stage T exit."

Re-run each item after T6+T7 land (BW sweep picks winner + retry timer
tightens), at which point SET delivery should go from "1-3 retries × 3s
each" to "< 1s typical" and these can be validated end-to-end.

## Re-validation trigger points

1. **After T6 lands (BW500 runtime config proven on hardware):** retry every
   `⚠` item above. The BW500/10Hz combo should have first-try ACK ≥ 60%
   per plan — enough to make single-SET testing reliable.
2. **After T7 lands (retry timer 3s → 500ms):** re-run with short retry
   window to confirm it doesn't introduce new edge cases.
3. **At T10 exit soak:** full fault-injection campaign against each ⚠ item.
   Specifically: station-ack-suppress fault-injector (`g_fault_station_ack_suppress`)
   can force the revert path; a power-cycle test after a SET+debounce cycle
   can confirm boot-read round-trip.

## What proceeding without this validation means

Risk: code paths that haven't run under real hardware may have subtle
timing/ordering issues. Mitigation: all ⚠ items are guarded by the same
ground-state + whitelist gates, so a bug can't land the vehicle in a
flight-unsafe state — worst case is a failed SET that reverts (which is
the intended safety net).
