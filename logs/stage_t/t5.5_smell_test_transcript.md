# IVP-T5.5 Preliminary Smell-Test — Council Transcript

**Date:** 2026-04-19
**Purpose:** Premise-level review of runtime radio config push techniques BEFORE the full plan was shown. User wanted fresh eyes on the core ideas without being biased by implementation details.
**Session type:** Smell-test (first of two). Correctness-council transcript is in
`logs/stage_t/t5.5_correctness_council_transcript.md`.
**Panelists:** NASA/JPL Avionics Lead, ArduPilot Core Contributor, Cubesat Startup Engineer, Advanced Hobbyist Rocketeer

## System context given to the panel

RocketChip is a hobbyist/educational flight-computer project on RP2350 MCUs. Ground
"station" (Fruit Jam) + vehicle "Feather" communicate via SX1276 LoRa 915 MHz with
configurable bandwidth/SF/CR/rate/power. Current radio config is compile-time fixed;
changing BW means reflashing both boards with matching firmware. Stage T needs to
sweep BW to pick the right operating point.

## The 4 core techniques smell-tested (premise only)

- **A.** Station-commanded runtime config push with set/ACK/swap confirmation and LOS
  watchdog auto-revert.
- **B.** Silence-based mismatch detector — dashboard warning when link has been
  silent too long.
- **C.** Manual channel-find scan — operator presses a key, station cycles through
  predefined configs listening for the vehicle.
- **D.** Current-config echo in vehicle downlink — nav packets include current
  radio config so station can display actual-vs-expected.

## Verdict

| Technique | Verdict | Reason |
|-----------|---------|--------|
| **A — Station-commanded set/ACK/swap** | ⚠ revise | Sound pattern, but: (1) apply must hook to **TxDone IRQ**, not a timer; (2) revert should be **symmetric** — vehicle also falls back after bounded silence; (3) validate IDLE at *apply*, not just at receive. |
| **B — Silence-based mismatch detector** | ⚠ revise | Good concept, but threshold must be **packet-count-based**, not wall-clock. Scales correctly across 2/5/10 Hz rates. |
| **C — Manual channel-find scan** | ✓ accept | Keep it passive (receive-only, no TX during scan), bounded linear sweep of a whitelist, dwell long enough for slowest profile. No cleverness. |
| **D — Current-config echo in downlink** | ⚠ revise | Needs an explicit **version/extension marker** (MAVLink v2 pattern or CCSDS secondary-header flag / APID). Without it, cross-version pairings produce phantom mismatches. |

## Recommended corrections for the plan

1. **Apply-after-TxDone**, not fixed 500 ms timer. Key the radio swap off the ACK's
   `TxDone` IRQ, with a 200 ms safety timer as backstop.
2. **Symmetric auto-revert.** Vehicle caches old config, reverts after 10 s if no
   valid RX on new config. Station reverts after 3 s visual, 12 s final.
3. **Packets-missed, not seconds,** for all silence thresholds. Scales across nav_hz range.
4. **MAV_CMD_USER_2 cast convention** documented in handler + unit test with
   125/250/500/7/8/9/10/11/12 edge values. Use `lroundf()`.
5. **New APID 0x101** for nav-with-embedded-config. Old APID 0x100 stays valid
   for backward compatibility.
6. **Scan list lives in `radio_config_table.h`.** Markdown doc references the
   header, not vice versa.
7. **Dashboard messages are specific and timed:** "Config push unconfirmed —
   reverting in 9s" with countdown, not generic "link silent."

## Panel additions (techniques not in original 4)

- **Explicit `QUERY_RADIO_CONFIG` command** — synchronous probe of vehicle's current
  config. Useful after a scan hit for authoritative confirmation. Separate MAV_CMD
  (e.g., MAV_CMD_USER_3).
- **"Config just changed" flag in first nav packet on new config** — UX confirmation
  that the transition physically happened. Distinguishes "I got a packet and maybe
  my old config was fine" from "I got a packet that was deliberately sent on the
  new config."

## Real-world systems cited by panel

- **CCSDS COP-1** — commanded parameter exchange, but has sequence counter we don't
  need right now.
- **ISS S-band/Ku-band handover** — closest pattern match; two-phase with fallback
  timer.
- **DSN scheduled channel swaps (MRO, MSL)** — scheduled-epoch model, good reference.
- **ExpressLRS Model-Match + rate tables** — packet-rate negotiation, revert-to-
  baseline on bind mismatch.
- **CRSF link-stats subpacket** — both ends broadcast agreed rate.
- **AX.25 SABM** — historical mode-change handshake.
- **GomSpace NanoCom U482x telemetry-lost patterns** — orbital practice.
- **SatNOGS client's decoder fallback** — "try N configs" pattern.

## Open questions flagged for correctness-council review

1. **Two-phase vs one-phase commit** — is the vehicle allowed to self-commit with a
   timer, or does ground-final-go belong here? NASA persona wanted to push harder.
2. **Second config-push during revert window** — needs explicit handling.
3. **In-flight safety race** — dispatch rejects non-IDLE, but what if state
   transitions during the pending-apply? Race condition between ARM and config-apply.
4. **Power-level asymmetry** — if station cranks vehicle TX power UP, next ACK is
   louder; if DOWN, station may lose confirming packet. Should power changes always
   be LAST-applied?
5. **What if vehicle's old-config cache is corrupted?** (e.g., power glitch during
   swap.) Need NVM-persisted known-good config as final fallback?

## Consensus

**Premise sound; revisions mechanical-not-structural.** Proceed to correctness
council with the full plan including these corrections folded in. No panelist
dissents on the overall design direction.
