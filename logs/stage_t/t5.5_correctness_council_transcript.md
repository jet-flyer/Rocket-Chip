### Round 2 — races and revert

**JPL:** Topic 2: second-command-during-revert. User description: station's LOS watchdog (~6 packets) fires before vehicle's (~15-20 packets). Station is already reverting to OLD. Operator sends another SET. What happens?

**ArduPilot:** Several branches. (a) Station has finished its own revert (radio now on OLD), so the new SET goes out on OLD. Vehicle is still on NEW and silent from station POV, but actually still applying its new-config watchdog. Vehicle's RX — remember, vehicle is in `kRxWindow` on NEW config — so vehicle doesn't hear the OLD-config SET either. Nothing happens. Watchdog expires, vehicle reverts to OLD on its own, and NOW the new SET lands (on OLD). Fine, bit slow, but correct. (b) Station is mid-revert — setters being applied one at a time. Operator presses `r`. This is the dangerous one: the station's `cmd_radio_config` handler needs to queue or reject. A naive implementation wires it up to fire a `SIG_RADIO_CONFIG_APPLY` into `AO_Radio` immediately, and now you have two applies racing: the revert path and the new SET path. In QV cooperative scheduling, the one that posts second wins, but intermediate state (mid-setter) is visible.

**Cubesat:** The right answer is a state machine in station's `AO_Telemetry` (or wherever `cmd_radio_config` lives). States: IDLE, PUSHING (ACK sent, LOS watchdog armed), CONFIRMED, REVERTING. Operator input while not IDLE is either queued (one deep) or rejected with a visible dashboard message. DSN does the first. Mission Control can set up a "next move" but nothing fires until the current move resolves.

**Rocketeer:** I vote reject with a visible dashboard message, not queue. An operator frantically mashing `r` in the field is a sign of panic, not a sign of "please carefully queue my intent." Show "Config push in progress, wait 15s" and refuse.

**JPL:** Consensus: reject, not queue. Add as a pass criterion. Topic 3: power-change-order.

**Cubesat:** Old NASA saying: "turn the light on before you try to read by it." If you raise power, do it last so the first confirming packet comes through at the new higher power. If you lower power, do it LAST as well so you don't deafen yourself before the confirming packet arrives. Either way — power is always the last setter called. This is one-line discipline in the apply handler.

**ArduPilot:** Stronger: power changes should be a separate, slower-path command. Getting power wrong is the only change that can permanently break the link outright. BW/SF/CR at worst gives you a retry loop. A 20→2 dBm power change can leave you at -120 dBm at the far end. Either (a) separate `MAV_CMD_USER_4 SET_RADIO_POWER` with its own confirmation cycle, or (b) bound power changes in the whitelist to ±6 dB per command.

**JPL:** I like (b). It's one line in the validator and it keeps the API surface small. A 20 → 14 dBm step is a 6-dB halving of EIRP, still closes a 2 km link by a wide margin per your T4 data. An operator who wants 20 → 2 must issue three commands, each confirmed, each at a known-workable link.

**Rocketeer:** Works for me. Set that as the whitelist constraint, make sure the help text shows what the next-valid powers are. "Current 20 dBm, next options: 14 or 17 or 20" on the `r` prompt.

**ArduPilot:** And still: power is last setter called in the apply handler. Defense in depth.

### Round 3 — NVM, packet counts, auto-mode

**JPL:** Topic 4: NVM fallback.

**Cubesat:** This one I need to push back on the plan. The plan says "deferred" with the rationale that power glitch = restart = boot to compile-time defaults. That's fine for MOST power glitches — the cap survives a sub-second dropout, the firmware boots clean, vehicle is on default config. That's fine too. The ACTUALLY scary case is: vehicle receives SET for a non-default operator-tuned config (say BW500 for range extension at a specific launch site), station and vehicle are both on BW500, a micro-brownout hits both, vehicle boots to defaults (BW125), station boots to defaults (BW125). Everyone's aligned. Operator doesn't notice they're back on slower config. Launches anyway. On the pad the collision window geometry is back to 140ms>100ms and the ACK rate is 6% again.

**JPL:** Two items I want to surface before we close. First: the plan treats the "first nav packet on new config has 'just changed' flag" as an independent UX signal. **Cubesat:** File this under "Red Flags for Implementer" rather than a new gate — if an audit shows the setters are clean, no change needed. If not, one more fix.

**Rocketeer:** Closing thought: scan list in `radio_config_table.h`. Plan says ≤10 configs. ## Recommended plan edits

**ADD to §3 IVP-T5.5 Scope constraints (between items 2 and 3):**

> **2a. Reinit-recovery config reapply (PREREQUISITE).** Extract the RadioConfig apply block at `ao_radio.cpp:277-291` into a helper `apply_runtime_config(s)` that operates on `RuntimeRadioConfig`. Call it from both `RadioAo_initial` and the reinit recovery path at `ao_radio.cpp:148-152`. Without this, any TX-fail-triggered reinit during or after config push silently reverts the radio to compile-time defaults while `RuntimeRadioConfig` still says NEW — a covert divergence.

**CHANGE §3 scope constraint 1 (flight-state gate) to also include whitelist validation:**

> Vehicle accepts `CMD_SET_RADIO_CONFIG` only in **IDLE or DISARMED** phase AND only if the 5-tuple (bw, nav, sf, cr, power) matches a row in `include/rocketchip/radio_config_table.h`. Validation via `lroundf()` from MAVLink floats, then whitelist lookup, then flight-state. Denied-ACK returned for any mismatch. Unit tests cover: boundary values 125/250/500/7/8/9/10/11/12, off-by-one float tolerance (249.9999f, 250.0001f both → 250 and accepted), invalid tuples (SF=0, CR=0, BW=999) denied.

**CHANGE §3 scope constraint 2 (protocol) — remove "config just changed flag":**

> Replace reference to "first nav packet emitted on new config sets a 'config just changed' flag bit" with: station confirms transition when it receives an APID 0x101 packet whose config echo field matches the last-sent SET payload. APID 0x101 alone is the versioning signal; no special-case first-packet flag.

**ADD new §3 scope constraint (station cmd state machine):**

> **Station command state machine:** `cmd_radio_config` and `cmd_query_config` run through a single-operator state {IDLE, PUSHING, CONFIRMED, REVERTING}. Only IDLE accepts new operator input. Other states display "Config push in progress, wait Xs (seconds until IDLE return)" on dashboard and refuse input. No command queuing.

**ADD to §3 power-change discipline (new scope constraint):**

> **Power-change ordering and bounding.** In the vehicle apply handler, setters fire in the order: bw → sf → cr → power (power always last). Power changes are bounded to ±6 dB per single SET. Whitelist entries with power jump >6 dB from current config are rejected with denied-ACK; operator must step through intermediate powers sequentially.

**CHANGE §3 silence detector thresholds:**

> Packet-count thresholds scale with `nav_rate_hz` with a floor: `station_thresh = max(6, ceil(1.5 * nav_hz))`, `vehicle_thresh = ceil(2.5 * station_thresh)`. At 2/5/10 Hz: station={6,8,15}, vehicle={15,20,38}. Vehicle threshold strictly greater than station's (asymmetric revert preserved).

**REMOVE from §3 scope constraint 10 (auto-reconfigure):**

Delete the whole scope constraint 10 body. Replace with:

> **Auto-reconfigure mode: DEFERRED** to a later stage. Manual `r`/`c`/`q` flow only in T5.5. Revisit after operational use in one or more flights / bench sessions to determine if auto adds value beyond a single-key `a` wrapper around existing commands.

**ADD to §3 dashboard (new row requirement):**

> **Station boot-banner config announce:** on station reboot, dashboard shows active radio config in high-contrast red text for first 30 seconds with prompt "Config: BW500 10Hz SF7 CR5 20dBm — [c]onfirm / [d]efault". Operator must confirm or reset. Prevents silent post-brownout revert-to-defaults scenario.

**CHANGE §3 pass conditions from 5 to 7:**

> 1. Unit tests pass: dispatch validator, whitelist lookup, float→int boundary cases, flight-state gate, denied-ACK emission.
> 2. QUERY round-trip: `q` → vehicle current config reply visible on station.
> 3. SET no-op: `r` with same-as-current config → full round-trip including TxDone apply and confirmation packet.
> 4. SET real change: BW change (e.g., 125 → 500) → full round-trip on both vehicle and station; dashboard shows new config.
> 5. Symmetric revert: kill station mid-cycle → vehicle auto-reverts to prev_config after its packet-count window; station scan re-finds on OLD.
> 6. **Reinit-recovery reapplies RuntimeRadioConfig:** force `kTxFailReinitThresh` during active push (e.g., detach radio antenna 3 TX cycles) → radio reapplies `RuntimeRadioConfig` post-reinit, not compile-time defaults. `RuntimeRadioConfig` and actual register state remain consistent.
> 7. Channel-find: `c` → operator finds vehicle at non-default config within ~2s scan window via CAD-accelerated sweep; CRC-verified source-ID match.

## Red flags for implementer

1. **The reinit-recovery path is a trap.** Any time `kTxFailReinitThresh` fires during testing, verify the radio ends up back on `RuntimeRadioConfig`, not compile-time defaults. Don't trust "usually works" — instrument and confirm.
2. **Audit each SX1276 setter for Standby→write→restore discipline.** Read `rfm95w_set_bandwidth/_spreading_factor/_coding_rate/_tx_power` in `rfm95w.cpp` before writing the apply handler. If any do raw register writes without mode transitions, fix that FIRST. Without it, the "setters-only" premise is unsafe.
3. **`r` and `q` key binding grep.** Confirm via `rc_os.cpp` grep that those keys are unallocated before binding. Plan assumes they are but the implementer must verify.
4. **Power setter is last in the apply chain.** Not "usually last" — audit code to confirm literal call order: `set_bandwidth`, `set_spreading_factor`, `set_coding_rate`, `set_tx_power`. Code review red-flag if power isn't dead last.
5. **Whitelist table must include at least one `nav_hz=2` entry and one BW500/10Hz entry.** Former for low-rate fallback; latter for T6-anticipated operational point. Missing either = operational hole.
