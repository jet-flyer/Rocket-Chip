## Deviation Acceptance Policy

**Default disposition for a finding is REMEDIATE, not accept.** We prefer to keep coding standards even in non-flight code for consistency and additional safety. A deviation lands in the Active section below only when remediation is either a major stumbling block or would take an unreasonable amount of time. Auto-generated code (with NASA SWE §8.11 (Auto-Generated Code) assurance practices) is the canonical accepted case. Findings that don't meet that bar go to the remediation queue.

Items expected here are rare. If the Active section grows beyond a handful of rows, it's a signal that the remediation queue is being neglected, not that the project is accumulating "acceptable" deviations.

**Note on P10 Rule 2 (loops with fixed upper bound):** Holzmann's original P10 paper explicitly exempts non-terminating iterations (e.g., RTOS scheduler, fault halt). For those, the inverted rule applies: it must be statically provable that the loop *cannot* terminate. We satisfy the inverted rule for our scheduler-class loops (QF_run, Core 1 sensor loop, fault halt). Those are not deviations — they are compliant under Holzmann's exemption — and therefore do not appear in this file. See `standards/CODING_STANDARDS.md` for the platform-conventions write-up.

### JSF AV Rule 182 — Pointer Casts in Native Code

JSF-182 ("Type casting from any type to or from pointers shall not be used") carries two exceptions: **Exc 1** — `void*`→`T*` via `static_cast` in low-level memory-management routines; **Exc 2** — literal hardware-address→pointer (rule's own example: `reinterpret_cast<Device_register>(0XFFA)`). The 2026-06-23 L2-P5 walk inventoried *every* `reinterpret_cast` in `src/` and dispositioned each individually (zero-exceptions policy — never auto-accept). **Compliant by Exc 2:** literal/HW-address→pointer (NVIC `0xE000E200`, `XIP_BASE`, PSRAM bases). **Remediated to compliance:** struct→`uint8_t*` CRC casts (CRC primitives take `const void*`; the single internal cast is Exc 1) and buffer→header overlays (`reinterpret_cast`→`static_cast`, Exc 1). **Centralized:** the QP/C `QEvt*`→derived-event downcast (~10 sites → one `static_assert(std::is_standard_layout_v<E>)`-guarded `evt_cast<E>` helper; well-defined via standard-layout first-member equivalence). The genuine native-code residual — banned by the rule with no applicable exception — is below.

| ID | Location | Rule | Severity | Difficulty | Rationale |
|----|----------|------|----------|-----------|-----------|
| CAST-1 | `src/main.cpp:210`, `src/core1/sensor_core1.cpp:483` — `mpu_setup_stack_guard(reinterpret_cast<uintptr_t>(&__StackBottom / &__StackOneBottom))` | JSF AV 182 (pointer→integer) | Accepted | N/A — hardware-interface | **The project's FIRST accepted deviation in native (project-authored) code — flagged here on crossing that threshold; repo-owner retains residual suspicion (2026-06-23) and a future re-examination is explicitly invited.** Programming the ARM MPU region-base (RBAR — a numeric register) requires a linker-defined stack-boundary symbol's *address as an integer*; standard C++ offers no cast-free way to obtain a linker symbol's address as a number. Rule 182's stated pointer→integer harm is truncation "if the pointer can represent values larger than the integral type to which it is converted" — impossible on RP2350 (32-bit pointers; `uintptr_t` is pointer-width *by definition* → provably lossless; hardened from `uint32_t` precisely so the no-loss property is self-evident). Sequestered to MPU / stack-guard boot setup. Primary source: JSF-AV-rules.pdf Rule 182 (Doc 2RDU00001 Rev C, p.55). **Independent check:** confirm `sizeof(void*) == sizeof(uintptr_t) == 4` on RP2350; confirm no standard means exists to read a linker symbol's address as an integer without the cast. |
| CAST-2 | ~10 QP/C event-dispatch sites, centralized in `evt_cast<E>()` (e.g. `src/active_objects/ao_notify.cpp:184` and the AO `dispatch` handlers) | JSF AV 182 (`QEvt*`→derived-event pointer) | Accepted | Hard — framework-intrinsic | QP/C carries derived events as `struct E { QEvt super; ... }` (first-member composition, **not** inheritance) and dispatches `QEvt const*`; reaching derived fields requires a `QEvt*`→`E*` downcast. **No JSF-182 exception covers this** (it is neither `void*`→`T*` nor literal→pointer) — the justification is *external to the ruleset*: the C++ standard guarantees a standard-layout object's address equals its first member's, so the cast is well-defined, and Rule 182's stated alignment-UB harm cannot occur (`QEvt` is correctly aligned at offset 0). ## Third-Party / Vendored Code (Adopted-Code Register)

Code we **don't own** (Pico SDK, QP/C, lwGPS, MAVLink, ETL, linker/CMSIS-supplied symbols) — "adopted code" per `CODING_STANDARDS.md` → "Adopted (Third-Party / Vendored / Toolchain) Code". Our coding standards are not re-derived onto this code; it is assured at the interface/integration/behavioral level. This register records *which* vendored constructs are accepted, and the evaluation behind each — so a green gate is not silently hiding an un-evaluated vendor flag (the "looks-covered-but-isn't" failure mode).

**Acceptance discipline — NEVER auto-accept.** A vendored construct (or a standards flag it raises) lands here ONLY after individual evaluation: **(1)** it is properly **sequestered / isolated**, and **(2)** it is **independently verified** that the rule's underlying *reason* no longer applies (or is satisfied at another level). Each entry below records that evaluation. An un-evaluated vendor flag is a finding, not an exemption. (A flag on **project-authored** code is never resolved here — it is a real finding handled as our own code; e.g., the same 2026-06-23 reserved-identifier sweep that produced TP-2 *also* found project-authored `_test_mode_*` names, which are remediated in code, not vendor-accepted.)

**Pre-commit hook scope:** the project's clang-tidy gate (`scripts/hooks/pre-commit`) checks staged files matching `src/*.cpp` only — it does not flag SDK or vendor headers. The deviations below are surfaced when running clang-tidy with `--header-filter=.*` and `--system-headers` (e.g., during a milestone audit).

### TP-2: Linker-supplied reserved identifier `__StackBottom`

**Source:** `__StackBottom` — a symbol **defined by the linker script** (ARM-GCC / pico-sdk), the start of the MPU stack-guard region. Referenced via `extern` in `src/main.cpp:82`, `src/safety/fault_protection.cpp:21`, `src/safety/fault_inject.cpp:152`.

**Standard:** `bugprone-reserved-identifier` (CERT DCL37-C / MISRA C 21.2) — "do not declare/define a reserved identifier."

**Evaluation (per the acceptance discipline):**
- **(1) Sequestered:** confined to the MPU/stack-guard subsystem — boot setup (`main.cpp` → `mpu_setup_stack_guard(&__StackBottom)`) and the fault handler / fault-injection test (`fault_protection.cpp`, `fault_inject.cpp`). *Follow-up hygiene (non-blocking): consolidate the 3 `extern` declarations into one boundary header (e.g. `include/rocketchip/linker_symbols.h`) so the reference exists in exactly one place.*
- **(2) Rule reason no longer applies — independently verified:** the rule prohibits *declaring/defining* a reserved name to avoid **colliding with or redefining** the implementation's symbol (UB). We do neither — `__StackBottom` is **defined by the linker**, and we only **reference** it (the intended use of a linker-published symbol); taking its address for the MPU guard. No collision, no redefinition, no UB. CERT DCL37-C's prohibition is on *creating* reserved names; referencing the toolchain's own symbol is the documented exception space. Precedent: Infineon PDL and Xen formally deviate MISRA 21.1/21.2 for exactly these `__`-prefixed toolchain symbols.

**Verdict:** accepted-via-vendoring (toolchain-supplied interface). **Do NOT rename** — it is the linker's name; renaming would break the MPU stack-guard linkage.

**Severity / Difficulty:** Accepted / N/A.

**Discovered:** 2026-06-23, L2-P5 §CM reserved-identifier measurement (WSL build). The *same* sweep flagged project-authored `_test_mode_*` names — those are **not** vendored and are remediated in code (Phase C), demonstrating the never-auto-accept discipline.
